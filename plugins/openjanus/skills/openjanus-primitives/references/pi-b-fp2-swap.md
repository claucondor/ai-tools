# pi_b Fp2 Swap — EIP-197 Coordinate Order

## The problem

snarkJS outputs the `pi_b` component of a Groth16 proof as a 2x2 matrix of BN254 Fp2 elements in `(re, im)` order:

```
pi_b = [[b00_re, b00_im], [b10_re, b10_im]]
```

The EIP-197 `ecPairing` precompile (and Solidity verifiers generated by `snarkjs zkey export solidityverifier`) expect coordinates in `(im, re)` order — reversed:

```
_pB = [[b00_im, b00_re], [b10_im, b10_re]]
```

Without the swap, `verifyProof` returns `false` for every valid proof. There is no revert, no error message — just a silent `false`.

## The fix

Apply `applyPiBSwap` from `@openjanus/sdk/utils` before any on-chain submission:

```typescript
import { applyPiBSwap } from "@openjanus/sdk/utils";

const { pA, pB, pC } = applyPiBSwap(rawSnarkProof);
// pB is now in EIP-197 order — safe to pass to verifyProof
```

`proveForEVM` and `verifyOnChain` in the SDK apply this automatically. Manual callers must apply it themselves.

## Why this exists

This is a known discrepancy between snarkJS's internal representation and the EVM encoding. The swap is documented in EIP-197 and in the snarkJS export calldata format. It applies to all Groth16 proofs on any EVM, not just Flow EVM.

## Diagnostic

If `verifyProof` returns `false` for a proof that passes local verification (`verifyLocally` returns `true`), the pi_b swap is the most likely cause. Run:

```typescript
import { applyPiBSwap } from "@openjanus/sdk/utils";
import { verifyOnChain } from "@openjanus/sdk/primitives";

// Without swap — will return false for valid proofs
// const valid = await verifier.verifyProof(proof.pi_a, proof.pi_b, proof.pi_c, pubSignals);

// With swap — correct
const { pA, pB, pC } = applyPiBSwap(proof);
const valid = await verifier.verifyProof(pA, pB, pC, pubSignals);
```
